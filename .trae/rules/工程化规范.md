# Rust 工程化规范

**版本**: v1.0 | **适用技术栈**: Axum + Dioxus + SQLx + PostgreSQL + Redis

---

## 一、核心工程化原则

| 原则 | 说明 |
|------|------|
| **行级TDD** | 每次只生成单行/单块极简代码，`cargo check` 分步校验 |
| **依赖管理** | 根 workspace 使用 `*` 版本号，features 在子 crate 按需开启 |
| **MSRV 约束** | 最低支持 Rust 1.96，工具链通过 rust-toolchain.toml 管理 |
| **零警告上线** | clippy `-D warnings`，fmt 检查通过后才能提交 |

## 三、依赖管理规范

### 3.1 根 workspace 配置策略

```toml
# Cargo.toml - workspace.dependencies
[workspace.dependencies]
# 根配置只保持版本号为 *，不设置 features
tokio = "*"
axum = "*"
tower-http = "*"
sqlx = "*"
redis = "*"
serde = "*"
```

### 3.2 子 crate 引用规范

```toml
# 子 crate 的 Cargo.toml 必须使用 .workspace = true + features 按需开启
[dependencies]
domain.workspace = true
application.workspace = true
tokio = { workspace = true, features = ["full"] }
axum.workspace = true
serde = { workspace = true, features = ["derive"] }
```

### 3.3 Features 按需编译原则

| 层级 | 职责 | 示例 |
|------|------|------|
| **根 workspace** | 仅定义依赖名称，使用 `*` 版本 | `tokio = "*"` |
| **子 crate** | 按需开启 features，精确控制 | `tokio = { workspace = true, features = ["full"] }` |

**规则**：
- 根 `Cargo.toml` 中**禁止**设置 features
- 所有 features 必须在**子 crate** 中按需开启
- 保持版本号为 `*` 以获取最新兼容版本

## 四、开发流程规范

### 4.1 新增业务模块流程

```
1. 定义领域模型    → domain/src/aggregate/
2. 创建出站端口    → domain/src/port/
3. 实现应用服务    → application/src/service/
4. 创建 DTO       → application/src/dto/
5. 实现数据库适配器 → database/src/
6. 创建 HTTP 控制器 → backend/src/api/
7. 注册路由       → backend/src/api/routes.rs
8. 配置依赖注入   → backend/src/di.rs
```

### 4.2 代码生成行级TDD流程

```
任务拆分 → 函数骨架 → 逐行实现 → cargo check → 单元测试 → 代码审查
```

### 4.3 提交前检查清单

- [ ] `cargo fmt --all` 格式化通过
- [ ] `cargo clippy -- -D warnings` 零警告
- [ ] `cargo test --all` 测试通过
- [ ] commit message 符合 Conventional Commits 规范

## 五、CI/CD 规范

### 5.1 GitHub Actions 必须任务

```yaml
jobs:
  check:     # 代码质量检查（必须先通过）
  msrv:      # MSRV 兼容性检查
  test:      # 集成测试
  build:     # 构建验证
```

### 5.2 MSRV 检查机制

- 通过 `rust-toolchain.toml` 定义 MSRV 版本
- CI 中使用 `dtolnay/rust-toolchain@<version>` 安装指定版本
- 每次依赖变更后必须验证 MSRV 兼容性

## 六、Docker 构建规范

### 6.1 多阶段构建要求

- 必须使用 `cargo-chef` 缓存依赖层
- 最终镜像使用 non-root 用户运行
- 包含健康检查（HEALTHCHECK）

### 6.2 镜像优化

```dockerfile
FROM rust:1.96-slim AS chef          # 依赖安装
FROM chef AS planner                 # 依赖分析
FROM chef AS builder                 # 代码编译
FROM debian:bookworm-slim AS runtime # 最终镜像
```

## 七、命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 类型/组件 | 大驼峰 | `UserOrder`, `HttpClient` |
| 变量/函数 | 蛇形 | `user_id`, `create_order` |
| 常量 | 大写蛇形 | `MAX_RETRY_COUNT` |
| 数据库表 | 蛇形 | `user_orders` |
| API 路由 | 蛇形 | `/api/v1/user-orders` |

## 八、错误处理规范

```rust
// 领域层使用 thiserror
#[derive(Debug, thiserror::Error)]
pub enum DomainError {
    #[error("无效的业务规则: {0}")]
    InvalidBusinessRule(String),
}

// 应用层使用 anyhow
#[derive(Debug, thiserror::Error)]
pub enum ApplicationError {
    #[error("用户不存在: {user_id}")]
    UserNotFound { user_id: String },
}
```

## 九、文档要求

| 元素 | 要求 |
|------|------|

| 复杂逻辑 | 必须注释算法或业务规则 |
