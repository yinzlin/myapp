# Rust 全栈开发流程规范

**版本**: v1.0 | **适用技术栈**: Axum + Dioxus + SQLx + PostgreSQL + Redis

---

## 一、开发任务拆解规范

### 1.1 任务层级结构

```
Sprint（迭代）
  └── Story（业务故事）
        └── Module（模块）
              └── Function（函数）
                    └── Code Block（代码块）
                          └── Line-level Task（行级子任务）
```

### 1.2 最小粒度：行级子任务

- 每次只实现一行代码或一个极简代码块
- 必须通过 `cargo check` 后才能继续下一行
- 每完成一个代码块立即标注进度

---

## 二、行级TDD开发流程

### 2.1 标准流程

```
[用户需求]
      │
      ▼
┌─────────────────┐
│  任务拆解        │  → 按层级结构拆分
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  创建函数骨架    │  → 空的函数签名
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  行级实现       │  → 逐行添加代码
│  cargo check    │  → 逐行校验
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  单元测试       │  → 骨架完成后编写
└────────┬────────┘
         │
         ▼
    [任务完成]
```

### 2.2 输出格式要求

每次输出必须包含：

```markdown
【当前行级子任务】
- 子任务描述
- 实现思路

【代码】
```rust
// 单行或极简代码块
```

【说明】
- 本行逻辑作用
```

### 2.3 禁止行为

- ❌ 一次性输出完整大函数
- ❌ 不校验直接跳过多行
- ❌ 绕过 `cargo check` 继续编码
- ❌ 在未完成的函数中添加无关逻辑

## 三、API 开发流程（前后端分离）

### 3.1 共享类型优先原则

```
1. 先定义 Domain 类型（domain crate）
2. 再定义 API DTO（application crate）
3. 然后实现后端 Handler（backend crate）
4. 最后实现前端 API 客户端（frontend crate）
```

### 3.2 API 开发 Checklist

- [ ] Domain 类型已定义并文档化
- [ ] API Request/Response DTO 已定义
- [ ] Backend Handler 已实现
- [ ] Route 已注册
- [ ] Frontend API 客户端已封装
- [ ] 前后端类型对应关系正确

---

## 四、数据库开发流程

### 4.1 仓储模式实现顺序

```
1. 定义出站端口（domain/src/port/）
      ↓
2. 实现仓储（database/src/{sqlite,postgres}/）
      ↓
3. 注册到依赖注入（backend/src/di.rs）
```

### 4.2 SQL 迁移规范

- 迁移文件命名：`YYYYMMDDHHMMSS_description.sql`
- 必须包含 `UP` 和 `DOWN` 脚本
- 提交前必须手动验证 SQL 语法

## 五、错误修复流程

### 5.1 Clippy 警告修复

```
cargo check 报错
      │
      ▼
┌─────────────────┐
│  定位当前行问题  │  → 不改动其他已通过代码
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  单行极简修复    │  → 只改一行
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  cargo check    │  → 验证修复
└─────────────────┘
```

### 5.2 编译错误修复

- 先阅读完整错误信息
- 定位错误行，不动其他已通过代码
- 使用最小修改原则
- 修复后立即验证

---

## 六、代码审查要点

### 6.1 AI 自检清单

- [ ] 无 `unwrap()` 除非明确安全
- [ ] 无 `unsafe` 除非必要
- [ ] 无硬编码配置
- [ ] 错误处理完善
- [ ] 公共 API 有文档注释
- [ ] 符合命名规范

### 6.2 提交前检查

```bash
just ci-check  # fmt + check + lint
```

## 七、新项目初始化流程

### 7.1 一键初始化命令

```bash
# 1. 克隆模板
git clone https://github.com/yinzlin/myapp.git new-project
cd new-project

# 2. 重置 Git
rm -rf .git
git init

# 3. 修改项目名称
# - 更新 Cargo.toml workspace.package
# - 更新 docker-compose.yml container_name
# - 更新 .env.example

# 4. 安装开发依赖
just setup

# 5. 验证环境
just ci-check
```

### 7.2 首次开发流程

```
1. 阅读 .trae/rules/ 下的所有规范
2. 确认开发环境正常
3. 创建特性分支
4. 按行级TDD流程开始开发
```
